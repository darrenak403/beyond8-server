# Use Case-07.07: Refund Processing

**Use Case ID:** UC-07.07

**Module:** Payment & Enrollment

**Actor chính:** Student, Admin

**Actor phụ:** VNPay Gateway, Email Service

**Mô tả ngắn:** Student yêu cầu refund và Admin xử lý trong vòng 7 ngày sau enrollment

**Trigger:** Student click "Request Refund" trên enrolled course

**Pre-condition:**

- Student đã đăng nhập
- Student đã enroll course (paid)
- Đăng ký ≤ 7 ngày
- Tiến độ < 10%
- Chưa hoàn thành quiz/assignment nào

**Post-condition:**

- Nếu đủ điều kiện: Refund request được tạo
- Admin nhận được notification
- Sau khi Admin xử lý: Payment refunded hoặc request bị từ chối

**Main Flow (Student Request Refund):**

1. Student navigate đến trang course đã enroll
2. Hệ thống kiểm tra điều kiện refund:
   - enrolled_at ≤ 7 ngày
   - progress_percentage < 10%
   - Không có quiz/assignment completed
3. Nếu đủ điều kiện: Hệ thống hiển thị button "Request Refund"
4. Student click "Request Refund"
5. Hệ thống hiển thị form yêu cầu refund:
   - **Lý do (dropdown):**
     - "Course không như mong đợi"
     - "Vấn đề kỹ thuật"
     - "Tìm được course tốt hơn"
     - "Lý do khác"
   - **Mô tả chi tiết (textarea):** Tối đa 500 ký tự
   - Button "Submit Request"
6. Student điền form và click "Submit Request"
7. Hệ thống tạo refund_request record:
   - refund_request_id
   - enrollment_id
   - user_id = student_id
   - course_id
   - amount (số tiền đã thanh toán)
   - reason (dropdown value)
   - description (textarea value)
   - status = "pending"
   - requested_at = thời điểm hiện tại
   - progress_at_request = progress_percentage hiện tại
8. Hệ thống gửi notification đến Admin
9. Hệ thống hiển thị confirmation: "Yêu cầu hoàn tiền đã được gửi. Chúng tôi sẽ xem xét và phản hồi sớm."
10. Kết thúc Use Case (Student)

**Main Flow (Admin Approve Refund):**

11. Admin navigate đến Admin Panel > Refund Requests
12. Hệ thống hiển thị danh sách refund requests với các cột:
    - Tên student
    - Course
    - Số tiền
    - Lý do
    - Ngày yêu cầu
    - Tiến độ %
    - Status
    - Actions (Approve/Reject buttons)
13. Admin click vào request để xem chi tiết:
    - Thông tin đầy đủ về student, course, payment
    - Lý do và mô tả chi tiết
    - Tiến độ học tập (%, videos watched, quizzes taken)
    - Ngày enrollment vs ngày request
14. Admin đánh giá và click "Approve"
15. Hệ thống hiển thị confirmation dialog: "Xác nhận hoàn tiền [amount] VND cho [student_name]?"
16. Admin click "Confirm"
17. Hệ thống trigger VNPay refund API:
    - Gửi refund request với transaction_id gốc
    - Amount = số tiền đã thanh toán
18. VNPay xử lý refund (có thể mất 3-7 ngày làm việc)
19. Hệ thống cập nhật refund_request:
    - status = "approved"
    - approved_by = admin_id
    - approved_at = thời điểm hiện tại
    - refund_transaction_id (từ VNPay)
20. Hệ thống cập nhật payment record:
    - status = "refunded"
    - refunded_at = thời điểm hiện tại
21. Hệ thống thu hồi enrollment:
    - enrollment.access_granted = false
    - enrollment.status = "refunded"
    - Không xóa enrollment (giữ lại cho audit trail)
22. Hệ thống gửi email đến Student:
    - Subject: "Yêu cầu hoàn tiền đã được chấp thuận"
    - Body: "Tiền sẽ được hoàn về tài khoản của bạn trong 3-7 ngày làm việc"
    - Thông tin refund: Amount, Course, Expected refund date
23. Hệ thống gửi in-app notification cho Student
24. Kết thúc Use Case (Admin Approve)

**Alternate Flow:**

- **AF1: Student không đủ điều kiện refund (>7 ngày)**

  - Tại bước 2: enrolled_at > 7 ngày
  - Hệ thống không hiển thị button "Request Refund"
  - Hiển thị message: "Hết thời hạn yêu cầu hoàn tiền (7 ngày)"
  - Kết thúc Use Case

- **AF2: Student không đủ điều kiện (tiến độ ≥10%)**

  - Tại bước 2: progress_percentage ≥ 10%
  - Hệ thống không hiển thị button "Request Refund"
  - Hiển thị message: "Không thể hoàn tiền do đã hoàn thành ≥10% khóa học"
  - Kết thúc Use Case

- **AF3: Student đã hoàn thành quiz/assignment**

  - Tại bước 2: Có quiz hoặc assignment đã completed
  - Hệ thống không hiển thị button "Request Refund"
  - Hiển thị message: "Không thể hoàn tiền do đã hoàn thành quiz/assignment"
  - Kết thúc Use Case

- **AF4: Admin Reject Refund**

  - Tại bước 14: Admin click "Reject" thay vì "Approve"
  - Hệ thống hiển thị form nhập lý do từ chối (required)
  - Admin nhập lý do và click "Confirm Reject"
  - Hệ thống cập nhật refund_request:
    - status = "rejected"
    - rejected_by = admin_id
    - rejected_at = thời điểm hiện tại
    - rejection_reason = lý do từ admin
  - Hệ thống gửi email đến Student:
    - Subject: "Yêu cầu hoàn tiền bị từ chối"
    - Body: Lý do từ chối + hướng dẫn liên hệ support nếu cần
  - Enrollment vẫn active, Student vẫn có quyền truy cập course
  - Kết thúc Use Case

- **AF5: VNPay refund API thất bại**

  - Tại bước 17: VNPay trả về error
  - Hệ thống ghi log lỗi, retry 3 lần với exponential backoff
  - Nếu vẫn thất bại:
    - Cập nhật refund_request: status = "processing_error"
    - Gửi notification đến Admin: "Refund cần xử lý thủ công"
    - Admin phải xử lý refund thủ công (banking transfer) và cập nhật status sau
  - Kết thúc Use Case

- **AF6: Student đã có refund request pending**
  - Tại bước 4: Đã tồn tại refund_request với status = "pending"
  - Hệ thống hiển thị: "Bạn đã gửi yêu cầu hoàn tiền. Đang chờ xử lý."
  - Không cho phép tạo request mới
  - Kết thúc Use Case
